# Provenance Guard — Planning

Provenance Guard is a backend service that a creative-sharing platform can plug
in to detect AI use. It checks whether submitted text was written by a human or generated by AI,
scores how confident it is, shows a plain-language label to readers, and lets
creators appeal. 

---

## Architecture

The system has two flows. The submission flow classifies text and returns a
label. The appeal flow lets a creator contest a decision and marks it under
review. Each arrow below is labeled with what passes between components.

### Flow 1: Submission (`POST /submit`)

```
 platform backend
        │  raw text  {content}
        ▼
 ┌──────────────────┐   over limit → 429
 │  Rate limiter    │ ─────────────────────────────▶ error response
 │ (Flask-Limiter)  │
 └────────┬─────────┘
          │  raw text (within limit)
          ▼
 ┌─────────────────────────────────────────────┐
 │  Detection pipeline (pipeline.py)            │
 │                                              │
 │   raw text ─┬─▶ Signal 1: LLM semantic       │
 │             │      (Groq)         ─┐          │
 │             ├─▶ Signal 2: Stylometry ├─ p_ai  │   each signal → p_ai ∈ [0,1]
 │             │      (variance, TTR)  │  per    │
 │             └─▶ Signal 3: Lexical    │ signal │
 │                    tells           ─┘         │
 │                                              │
 │   3 signal p_ai ─▶ weighted ensemble         │
 │                    → combined p_ai,          │
 │                      confidence, verdict     │
 └───────┬──────────────────────────────────────┘
         │  p_ai + confidence
         ▼
 ┌──────────────────┐   verdict + confidence → label text
 │ Label builder    │
 │ (labels.py)      │
 └───────┬──────────┘
         │  {verdict, confidence, p_ai, signals[], label}
         ├───────────────────────────────┐
         │  full decision record          │  response payload
         ▼                                ▼
 ┌──────────────────┐              platform backend
 │ Audit store      │  append classification row (JSONL)
 │ (store.py,SQLite)│
 └──────────────────┘
```

### Flow 2: Appeal (`POST /appeal`)

```
 creator
    │  {submission_id, reasoning}
    ▼
 ┌──────────────────┐   missing id/reasoning → 400
 │ Appeal handler   │   unknown submission_id → 404
 │ (app.py)         │
 └────────┬─────────┘
          │  validated submission_id + reasoning
          ▼
 ┌──────────────────────────────────────────────┐
 │ Audit store (store.py, SQLite)                │
 │   • insert appeal row (links to submission)   │
 │   • UPDATE submission.status = 'under_review' │
 │   • append appeal entry to audit_log next to  │
 │     the original classification (JSONL)       │
 └────────┬─────────────────────────────────────┘
          │  {status: 'under_review', reasoning}
          ▼
       creator (response)
```

What each component does:

- Rate limiter: gates `/submit`.
- pipeline.py: runs the three signals and the weighted-average ensemble.
- labels.py: turns a score into text a reader sees.
- store.py: owns the SQLite tables and the append-only JSONL audit log.
- app.py: wires up the endpoints.

## Detection Signals

The pipeline uses three signals. This meets the ensemble stretch. Each signal
returns a probability `p_ai ∈ [0,1]`, where higher means more likely AI. The
output is a score, not a binary flag. Each signal returns an object like
`{p_ai: 0.72, features: {...}}`, where `features` holds the raw measurements
behind the score. So the individual signal scores show up next to the combined
score.

### Signal 1: LLM semantic classifier (Groq `llama-3.3-70b-versatile`)

- What it measures: meaning and style as a whole. Does the argument develop the
  way a person's would? Is there a real voice and some quirk? Or does the
  phrasing feel averaged? The model returns a probability and a short reason.
- Why we chose it: it is the only signal that reads meaning, not surface stats.
  It catches AI text that varies its sentences but stays bland underneath.
- What it misses: it is not deterministic. It can lean toward calling any
  polished writing AI, which hurts skilled humans. Light hand-editing of AI text
  can beat it. It is sensitive to how the prompt is worded.

### Signal 2: Stylometric variability

- What it measures: burstiness and vocabulary diversity. Burstiness is the
  coefficient of variation of sentence length (stdev / mean). Diversity is the
  type-token ratio (unique words / total words). Humans mix long and short
  sentences and use varied words. AI tends to stay uniform and repeat itself.
- Why we chose it: it is purely structural and does not care about meaning. So it
  can disagree with the LLM. That disagreement is what makes the ensemble useful.
- What it misses: it is unstable on short texts (under about 5 sentences). It is
  genre-blind, so formal or technical human writing looks AI-ish. Poetry and
  experimental prose break its assumptions.

### Signal 3: Lexical regularity 

- What it measures: surface fingerprints. Punctuation density (mainly em-dash
  overuse), function-word ratio, back-to-back n-gram repetition, and how often
  stock AI phrases show up ("moreover," "it's important to note," "in today's
  world," "delve," etc).
- Why we chose it: it is cheap and easy to read. And it catches a different
  failure mode than the other two: the phrasing signature of instruction-tuned
  models.
- What it misses: it is easy to evade with paraphrase or find-replace. It throws
  false positives on formal or ESL human writing that uses these connectives for
  real.

### Combining signals into a confidence score

Each signal gives a `p_ai ∈ [0,1]`. The ensemble is a fixed weighted average:

```
p_ai = 0.50 · LLM  +  0.30 · Stylometry  +  0.20 · Lexical
```

- LLM gets 0.50. It is the only signal that reads meaning, and it holds up best
  in practice.
- Stylometry gets 0.30. Good structural evidence, but fragile across genres, so
  it gets less than the LLM.
- Lexical gets 0.20. It is the weakest and easiest to evade, so it gets the
  least. It mostly breaks ties.

The weights are constants in `pipeline.py` and the API echoes them back. So a
reader can always see how a verdict was reached. Signals do not veto each other.
They average, and that is how conflicts get resolved.

Confidence is how sure we are of the verdict, not the raw AI-probability:

```
confidence = max(p_ai, 1 − p_ai)      # ranges 0.5 … 1.0
```

So `p_ai = 0.50` gives `confidence = 0.50`, a coin flip. And `p_ai = 0.95` gives
`confidence = 0.95`, near-certain it's AI. This is the meaning the project hint
asks for: 0.5 means we genuinely do not know.

## Uncertainty Representation

The verdict and label come from the ensemble `p_ai` using an asymmetric band.
Falsely accusing a human is worse than missing AI text (see the hint and Known
Limitations). So the bar to call something AI is higher than the bar to call it
human:

| Ensemble `p_ai`      | Verdict              | Label variant         |
|----------------------|----------------------|-----------------------|
| `p_ai ≥ 0.55`        | Likely AI-generated  | High-confidence AI    |
| `0.35 < p_ai < 0.55` | Uncertain            | Uncertain             |
| `p_ai ≤ 0.35`        | Likely human-written | High-confidence human |

These thresholds were calibrated in Milestone 4, not guessed up front. The first
plan used `p_ai ≥ 0.80` for the AI bar. But a weighted average of three noisy
signals compresses toward the middle. In testing, even blatant AI text topped out
near 0.69, so the 0.80 bar was never reached and nothing was ever called AI. The
real separation point between AI and human writing sits near 0.55, so that is the
AI bar now. See the README Spec Reflection for the full story.

The band stays asymmetric on purpose. The AI bar (0.55) leaves a wide grey zone
below it, so borderline work falls to Uncertain instead of being accused. The
human bar (0.35) is lower, so we only call something human when the signals
agree. Confidence is `max(p_ai, 1 − p_ai)`, so the human side tends to report
higher confidence than the AI side, which fits the goal of being careful before
accusing.

What a confidence of 0.6 means: the system leans one way but is not sure. A
`confidence = 0.6` comes from `p_ai` near 0.6 or 0.4. At 0.6 it just crosses into
Likely AI-generated. At 0.4 it stays Uncertain. Either way the number is close to
a coin flip, so the label wording is cautious. The label flips at the band edges,
not at a single 0.5 line, so a weak result and a strong result show different
text, not just a different number.

How we map raw outputs to the score: each signal turns its raw features into a
`p_ai` using documented heuristic thresholds (see Detection Signals). We do not
run formal statistical calibration. Instead we tune those per-signal thresholds,
the ensemble weights, and the band edges against the reference inputs below until
the scores separate clearly.

How we check the scores mean something: we run a set of reference inputs through
the pipeline and confirm they land in different bands. This lives in the repo as
`tests/validate_scoring.py`.

- A clearly AI-sounding paragraph. Expected: high `p_ai`, AI label.
- An idiosyncratic human excerpt with irregular sentences and personal voice.
  Expected: low `p_ai`, human label.
- A formal but human paragraph, and a lightly edited AI paragraph. Expected: both
  land mid-range, Uncertain.

If the AI and human cases do not produce visibly different labels and scores,
then the scoring is not showing uncertainty, and we retune the thresholds or
weights.

## Transparency Label Variants

These labels show to a reader on the platform. Plain language, no jargon. The
text changes with confidence, not just the number.

High-confidence AI (`p_ai ≥ 0.55`)
> Likely AI-generated. Our checks lean toward this piece being written with
> generative AI (confidence: {confidence}%). This is an automated estimate, not a
> certainty — the creator can appeal if they believe it's wrong.

High-confidence human (`p_ai ≤ 0.35`)
> Likely human-written. Our checks found no strong signs of AI generation
> (confidence: {confidence}%). This is an automated estimate and not a guarantee
> of authorship.

Uncertain (`0.35 < p_ai < 0.55`)
> Not enough signal to tell. Our checks were mixed and we can't confidently say
> whether this was written by a person or AI (confidence: {confidence}%). We're
> showing this openly rather than guessing.

## Appeals Workflow

Who can appeal: the creator of a submission. They appeal by sending the
submission id and their reasoning. What they provide: the `submission_id` and a
non-empty `reasoning` string.

1. A creator disagrees with a verdict and calls `POST /appeal` with the
   submission id and their written reasoning.
2. The handler checks that the submission exists and that the reasoning is not
   empty.
3. It records an appeal row (submission id, reasoning, timestamp) linked to the
   original decision. It writes an `appeal` entry to the audit log next to the
   original classification entry. And it sets the submission's status to
   `under_review`.
4. The response confirms the new status and echoes the logged appeal.

Automated re-classification is out of scope. An appeal flags the item for a human
reviewer.

What a reviewer sees in the queue: the queue is the set of submissions with
status `under_review`. For each item the reviewer sees the original content, the
verdict, the confidence, the combined `p_ai`, the three per-signal scores, and
the creator's reasoning, all tied together by submission id with timestamps. They
read this from the audit log (`GET /log`), where the original classification
entry and the appeal entry sit next to each other.

### Anticipated edge cases

1. Appeal for a submission id that doesn't exist or was mistyped. The handler
   returns `404`. It does not create a dangling appeal row or audit entry.
   Otherwise the log fills with orphan appeals that can't be tied to a decision.
2. Empty or whitespace-only reasoning. The rubric requires capturing the
   creator's reasoning. So a blank appeal is rejected with `400` instead of
   logged as a valid contest with no content.
3. Repeated appeals on an item that is already `under_review`. Each appeal is
   still logged, since creators may add new reasoning. But the status stays
   `under_review`. Re-appealing does not reset or escalate anything, and every
   attempt stays individually visible in the audit log.
4. Very short submission, like a haiku or one sentence. The stylometry and
   lexical signals are unstable here. For text under a minimum length the
   pipeline treats it as thin evidence and shrinks the combined score toward
   0.5, which pulls short text toward Uncertain. A short casual note can still
   land human, but a short AI-ish note gets pulled back so a creator isn't
   accused on a sentence or two.

## AI Tool Plan

We use AI coding tools (Claude Code) to generate code at the three
implementation milestones. For each one, we list the spec sections we hand to
the tool, what we ask it to generate, and how we verify the output.

### M3 — Submission endpoint + first signal

- Spec sections provided: the Architecture section (both flow diagrams) and the
  Detection Signals section, focused on Signal 1 (LLM semantic classifier).
- What we ask for: a Flask app skeleton with the `POST /submit` route, plus the
  first signal function that takes text and returns `{p_ai, features}`.
- How we verify: call the signal function directly on a few sample texts and
  check the scores look right before wiring it into the endpoint.

### M4 — Remaining signals + confidence scoring

- Spec sections provided: Detection Signals (Signals 2 and 3), the Uncertainty
  Representation section (thresholds and the confidence formula), and the
  diagram.
- What we ask for: the stylometry and lexical signal functions, plus the
  ensemble scoring logic that combines all three into `p_ai`, `confidence`, and a
  verdict. (The course path stops at two signals; the third is the ensemble
  stretch and gets added here.)
- How we verify: run clearly AI text and clearly human text through the pipeline
  and confirm the scores differ in a meaningful way, not a coin flip at 0.5.

### M5 — Production layer

- Spec sections provided: the Transparency Label Variants, the Appeals Workflow,
  and the diagram.
- What we ask for: the label generation logic that maps `p_ai` to a variant and
  fills in the confidence string, plus the `POST /appeal` endpoint, the SQLite
  store, and the audit log.
- How we verify: confirm all three label variants are reachable with different
  inputs, and confirm an appeal sets status to `under_review` and lands in the
  audit log next to the original decision.

We check every generated component against the spec sections that seeded it. We
record any divergences in the README's Spec Reflection.

## Data / Storage sketch

SQLite (`store.py`) with three tables:

- `submissions` — id, content, verdict, p_ai, confidence, status (`classified`
  or `under_review`), created_at.
- `appeals` — id, submission_id (FK), reasoning, created_at.
- `audit_log` — append-only rows. `GET /log` renders them as JSONL. Fields:
  timestamp, event (`classification` or `appeal`), submission_id, verdict,
  confidence, probability_ai, signals (per-signal scores), and appeal reasoning
  on appeal events.
