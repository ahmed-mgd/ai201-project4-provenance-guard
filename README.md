# Provenance Guard

A backend service that a creative-sharing platform can plug in. It checks whether
submitted text was written by a human or generated by AI, scores how confident it
is, shows a plain-language label to readers, and lets creators appeal.

AI detection is not a solved problem. So this system is built to report
uncertainty honestly, not to pretend it is always right. Its safest choice is to
say "not enough signal to tell" rather than wrongly accuse a human writer.

The full design doc is in [planning.md](planning.md), written before the code.

| Method + path | Purpose |
|---|---|
| `POST /submit` | Classify text. Body: `{text, creator_id}`. Rate-limited. |
| `POST /appeal` | Contest a result. Body: `{content_id, creator_reasoning}`. |

| `GET /health` | Liveness check. |

## Architecture overview — the path of one submission

A platform sends text to `POST /submit`.

1. The request hits the rate limiter. If the caller is over the limit it gets a
   `429` and stops here.
2. The text goes into the detection pipeline, which runs three independent
   signals. Each returns a probability that the text is AI-generated
   (`p_ai`, from 0 to 1).
3. The pipeline combines the three scores into one `p_ai` using a fixed weighted
   average, then derives a `confidence` and an `attribution`
   (`likely_ai` / `uncertain` / `likely_human`).
4. The label builder turns that result into one of three plain-language labels a
   reader would see.
5. The store saves the submission to SQLite and appends a structured entry to the
   audit log (JSONL).
6. The response returns the attribution, confidence, each signal's score, and the
   label text.

Appeals take a second path. A creator calls `POST /appeal` with the `content_id`
and their reasoning. The store records the appeal, flips the submission's status
to `under_review`, and writes an appeal entry to the audit log next to the
original decision. There is no automated re-classification; an appeal flags the
item for a human reviewer.

The full ASCII diagram of both flows is in
[planning.md → Architecture](planning.md#architecture).

## Detection signals

The pipeline uses three signals. Two are required; the third is the ensemble
stretch feature (see [Ensemble detection](#ensemble-detection-stretch-feature)
below). Each returns a probability `p_ai` and its own raw features, so individual
scores show up next to the combined score.

### Signal 1 — LLM semantic classifier (Groq `llama-3.3-70b-versatile`)

- What it measures: meaning and style as a whole. Does the writing have a real
  voice and specific lived detail, or does it read as smooth and averaged? The
  model returns a probability and a short reason.
- Why we chose it: it is the only signal that reads meaning, not surface
  statistics. It catches AI text that varies its sentences but stays bland.
- What it misses: it is not deterministic, so the same text can score a bit
  differently run to run. It leans toward calling any polished writing AI, which
  hurts skilled and formal human writers. Light hand-editing of AI text can beat
  it.

### Signal 2 — Stylometric variability (pure Python)

- What it measures: two structural numbers. Burstiness is the coefficient of
  variation of sentence length (stdev / mean). Diversity is the type-token ratio
  (unique words / total words). Humans mix long and short sentences and use
  varied words; AI trends uniform and repetitive.
- Why we chose it: it is purely structural and ignores meaning, so it can
  disagree with the LLM. That disagreement is what makes the ensemble useful.
- What it misses: it is unstable on short text. It is genre-blind, so formal or
  technical human writing reads as AI-ish. Poetry and experimental prose break
  its assumptions.

### Signal 3 — Lexical regularity and AI "tells" (pure Python)

- What it measures: surface fingerprints. How often stock AI phrases appear
  ("delve," "moreover," "it's important to note," "tapestry"), em-dash density,
  and back-to-back repeated word pairs.
- Why we chose it: it is cheap and easy to read, and it catches the phrasing
  signature of instruction-tuned models, which is a different failure mode than
  the other two.
- What it misses: it is easy to evade by paraphrasing. It false-positives on
  formal or non-native human writing that uses these connectives for real.

## Confidence scoring

### How the signals combine

Each signal returns a `p_ai` from 0 to 1. The ensemble is a fixed weighted
average:

```
p_ai = 0.50 · LLM  +  0.30 · Stylometry  +  0.20 · Lexical
```

- LLM gets the most weight (0.50). It is the only signal that reads meaning and
  is the most reliable.
- Stylometry gets 0.30. Good structural evidence, but fragile across genres.
- Lexical gets 0.20. The weakest and easiest to evade, so it mostly breaks ties.

The signals do not veto each other. They average, and that is how a conflict
between them is resolved. The weights are echoed back in every API response and
audit-log entry, so a reader can always see how a verdict was reached.

`confidence` is how sure we are of the verdict, not the raw AI-probability:

```
confidence = max(p_ai, 1 − p_ai)      # ranges 0.5 to 1.0
```

So `p_ai = 0.50` gives confidence 0.50 (a coin flip) and `p_ai = 0.95` gives
confidence 0.95. A value near 0.5 means we genuinely do not know.

### Thresholds and the three categories

| Combined `p_ai` | Attribution | Label |
|---|---|---|
| `p_ai ≥ 0.55` | `likely_ai` | High-confidence AI |
| `0.35 < p_ai < 0.55` | `uncertain` | Uncertain |
| `p_ai ≤ 0.35` | `likely_human` | High-confidence human |

The band is asymmetric on purpose. Wrongly accusing a human is worse than missing
AI text, so the grey "uncertain" zone sits below the AI bar and absorbs
borderline cases instead of accusing them.

### How we validated the scores are meaningful

We run a set of reference inputs through the pipeline and confirm they land in
different bands. This lives in the repo as
[`tests/validate_scoring.py`](tests/validate_scoring.py). Run it with
`python -m tests.validate_scoring`. Actual output:

| Input | LLM | Stylometry | Lexical | Combined `p_ai` | Confidence | Result |
|---|---|---|---|---|---|---|
| Clearly AI (marketing prose) | 0.80 | 0.184 | 0.60 | 0.575 | 0.575 | `likely_ai` |
| Clearly human (casual review) | 0.20 | 0.00 | 0.00 | 0.10 | 0.90 | `likely_human` |
| Borderline: formal human (econ) | 0.80 | 0.287 | 0.00 | 0.486 | 0.514 | `uncertain` |
| Borderline: lightly edited AI | 0.70 | 0.182 | 0.171 | 0.439 | 0.561 | `uncertain` |

The scores separate, and they are not a constant.

### Two submissions with noticeably different confidence

- High-confidence case — the casual restaurant review scored `p_ai = 0.05`,
  confidence `0.95`, `likely_human`. All three signals agreed: casual voice, high
  sentence variety, no AI phrases.
- Lower-confidence case — the formal economics paragraph scored `p_ai = 0.486`,
  confidence `0.514`, `uncertain`. Here the signals disagreed: the LLM thought it
  was AI (0.80) because it reads polished, but stylometry (0.287) and lexical
  (0.00) pulled it back. The ensemble reported honest uncertainty instead of
  accusing a likely human.

That gap — 0.95 versus 0.51 confidence — is the point. A confident case and an
uncertain case produce different numbers and different label text.

## Transparency label

The label is what a reader sees on the platform. It uses plain language, shows
the confidence as a percentage, and changes text with the verdict. The
`{confidence}%` placeholder is filled at runtime.

High-confidence AI (`p_ai ≥ 0.55`)
> Likely AI-generated. Our checks lean toward this piece being written with
> generative AI (confidence: {confidence}%). This is an automated estimate, not a
> certainty — the creator can appeal if they believe it's wrong.

High-confidence human (`p_ai ≤ 0.35`)
> Likely human-written. Our checks found no strong signs of AI generation
> (confidence: {confidence}%). This is an automated estimate and not a guarantee
> of authorship.

Uncertain (`0.35 < p_ai < 0.55`)
> Not enough signal to tell. Our checks were mixed and we can't confidently say
> whether this was written by a person or AI (confidence: {confidence}%). We're
> showing this openly rather than guessing.

All three are reachable. In the audit-log sample below, the same three submissions
produced `high_confidence_ai` (57%), `high_confidence_human` (95%), and
`uncertain` (51%).

## Appeals workflow

A creator who disagrees with a verdict calls `POST /appeal` with the `content_id`
and a `creator_reasoning` string. The endpoint:

1. Checks the submission exists (404 if not) and that the reasoning is not empty
   (400 if it is).
2. Records the appeal, sets the submission's status to `under_review`, and writes
   an appeal entry to the audit log next to the original classification.
3. Returns a confirmation with the new status.

Example:

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "d2e984b4-190e-468f-a51f-7bb3568c40ac",
       "creator_reasoning": "I wrote this myself. I am a non-native English speaker and my style may look formal."}'
```

Response:

```json
{
  "content_id": "d2e984b4-190e-468f-a51f-7bb3568c40ac",
  "status": "under_review",
  "message": "Appeal received. This submission is now under review.",
  "appeal_reasoning": "I wrote this myself. I am a non-native English speaker and my style may look formal.",
  "original_attribution": "likely_ai"
}
```

There is no automated re-classification. The appeal flags the item for a human
reviewer, who reads the original decision and the reasoning together from the log.

## Rate limiting

Applied with Flask-Limiter, keyed by client IP.

| Endpoint | Limit |
|---|---|
| `POST /submit` | 10 per minute, 100 per day |
| `POST /appeal` | 20 per hour |

Reasoning, tied to how a writing platform is actually used:

- A real creator submits their own work. They write for a while, then submit. Even
  a busy writer posts a handful of pieces a day and almost never more than one in
  the same minute. 10 per minute leaves generous room for normal bursts (editing
  and resubmitting, re-checking a piece) while stopping a script that hammers the
  endpoint.
- 100 per day caps one IP far above any genuine creator but low enough to stop a
  scraper from running thousands of classifications to reverse-engineer the
  detector. Every `/submit` also makes a paid Groq call, so the daily cap protects
  cost too.
- Appeals are rare, deliberate human actions. 20 per hour is plenty for a
  frustrated creator contesting several items, but it blocks appeal spam.

### Rate limiting in action

Sending 12 rapid requests against the 10-per-minute limit:

```
req 1 -> 200
req 2 -> 200
...
req 10 -> 200
req 11 -> 429
req 12 -> 429
```

The `429` body:

```json
{
  "error": "Rate limit exceeded. Please slow down and try again later.",
  "detail": "10 per 1 minute"
}
```

(append-only). `GET /log` returns it as JSON. Classification entries record the
timestamp, content id, attribution, confidence, all three individual signal
scores, the weights, and the label variant. Appeal entries record the reasoning
and the status change, linked to the original by `content_id`.

## Ensemble detection (stretch feature)

The system uses three distinct signals, not two, which is the ensemble stretch.

- The signals are genuinely different: one semantic (LLM), one structural
  (stylometry), one lexical (AI tells). See [Detection signals](#detection-signals).
- The weighting is documented and fixed: `0.50 · LLM + 0.30 · Stylometry +
  0.20 · Lexical`. Weights reflect how much we trust each signal.
- Conflicts are resolved by the weighted average — no single signal can veto the
  others. The formal-economics case above is a live example: the LLM said AI, the
  structural signals said human, and the average landed on `uncertain`.
- Individual signal scores appear next to the ensemble result in both the
  `/submit` response and the audit log, so you can always see the breakdown.

## Known limitations

The clearest failure case is formal writing by non-native English speakers and
academics. Two of the three signals are biased against it. The LLM signal leans
toward calling polished, evenly-structured prose AI — in testing, a formal
economics paragraph scored 0.80 from the LLM. Stylometry also reads low-burstiness
formal writing as AI, because formal registers use even, measured sentences. Since
two of three signals share this bias, the ensemble cannot fully correct it, and
this kind of genuine human writing gets pushed toward `uncertain` or, in stronger
cases, `likely_ai`. That is exactly the appeal in the sample log: a non-native
speaker's formal paragraph was flagged, and their appeal explains the mismatch.
This is the false-positive the asymmetric threshold and the appeals workflow exist
to soften, but the underlying signal bias is real.

Two related weak spots: poetry (heavy repetition and simple vocabulary drop the
type-token ratio, so stylometry reads it as AI) and lightly edited AI text (a few
human edits remove the lexical tells and can flip the LLM, so it lands mid-range).

## Spec reflection

How the spec helped: writing the three label variants and the threshold table in
planning.md before any code meant the label builder was a direct mapping. There
was no guesswork at build time — `labels.py` just fills the confidence percentage
into text that already existed. The architecture diagram did the same for module
layout: `signals/`, `pipeline.py`, `labels.py`, and `store.py` came straight from
the boxes in the diagram.

How the implementation diverged: the AI threshold. The plan set `p_ai ≥ 0.80` for
a `likely_ai` verdict. That number was a guess made before seeing real scores.
When I tested the pipeline in Milestone 4, a weighted average of three noisy
signals compressed toward the middle — even blatant AI text topped out near 0.69.
So the 0.80 bar was never reached and nothing was ever labeled AI. I recalibrated
the bar to 0.55 against the reference inputs, which is where AI and human writing
actually separate, and updated planning.md to match. A second, smaller divergence:
the plan said short text would "down-weight structural signals," but that made
short text more confident, not less, so I changed it to shrink the score toward
0.5 instead.

## AI usage

I used Claude Code as the AI tool, prompted with sections of planning.md as
described in the AI Tool Plan.

Instance 1: detection pipeline. I gave it the Detection Signals section and the
architecture diagram and asked it to generate the three signal functions and the
weighted-average combiner. It produced working signal code and a combiner that
used my planned `0.80` AI threshold and a short-text rule that boosted the LLM
weight. I overrode both. Testing showed 0.80 was unreachable, so I recalibrated to
0.55. And I found the short-text rule made tiny inputs more confident, not less —
a 3-word phrase scored `likely_ai` at 69% — so I replaced it with a shrink toward
0.5 that pulls short text to `uncertain`.

Instance 2: labels and appeals. I gave it the Transparency Label Variants and
Appeals Workflow sections and asked for the label builder and the `POST /appeal`
endpoint. It produced both, and I verified all three label variants against the
spec text. I revised one thing: the AI label first said the checks "strongly
suggest" AI. Because the calibrated AI verdict fires at moderate confidence
(around 57%), "strongly suggest" overclaimed, so I changed it to "lean toward,"
which matches what the confidence number actually says.
